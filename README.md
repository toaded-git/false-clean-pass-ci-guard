# false-clean-pass-ci-guard

`false-clean-pass` is a stateless GitHub Action for PRs that look green while CI trust was weakened. It reports skipped or empty tests, ignored failures, coverage and test-count drops, baseline changes, missing env declarations, and required-job skip-as-success risks in one required check.

## Organization Evidence Reports Are Self-Attested

The Organization Evidence Report is a SELF-ATTESTED evidence input for internal governance visibility. It is NOT an independent audit attestation, NOT audit evidence by itself, and NOT proof of control implementation. Treat the SOC2 / ISO mapping as a pointer for reviewers who will independently verify the underlying records.

The report is generated by the customer organization's own evidence-collection repo as a scheduled or manually dispatched batch. It is a static JSON file plus Markdown render. There is no database and no always-on service.

The license model is an honor-system OSS organization license in the same practical family as Gitleaks/keygen-style licensing. Runtime `owner.type` checks and local license gates are compliance signals, not a technical wall.

## Consumer Workflow

Use `false-clean-pass` as the required status check name in branch protection.

```yaml
name: false-clean-pass
on:
  pull_request:
    types: [opened, synchronize, reopened, labeled, unlabeled]

permissions:
  contents: read
  pull-requests: write
  checks: write
  security-events: write

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm test -- --reporter=junit --outputFile=junit.xml

  verify:
    runs-on: ubuntu-latest
    needs: [test]
    if: always()
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: toaded-git/false-clean-pass-ci-guard@v2
        env:
          FCP_LICENSE: ${{ secrets.FCP_LICENSE }}
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          requiredJobs: test,lint,false-clean-pass
          test-results-glob: junit.xml
          coverage-summary: coverage/coverage-summary.json
          ci-env-keys: DATABASE_URL,JWT_SECRET
          fail-on: error
          sarif-path: false-clean-pass.sarif
          evidenceOutput: fcp-evidence.json
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: false-clean-pass.sarif
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: fcp-evidence-${{ github.event.pull_request.number }}-${{ github.event.pull_request.head.sha }}
          path: fcp-evidence.json
```

Run the guard in its own job with `if: always()`. If the test job fails or is skipped, the guard still runs and records why the PR is not a clean pass.

## Evidence Records And License Markers

Every run can write an Evidence Record JSON artifact. Personal repositories and unlicensed organization repositories still get full detector output with `license.org=false` and `signature=null`.

When an organization repository provides a valid `FCP_LICENSE`, the record gets a license-holder org proof marker. That marker copies the issuer signature over the license payload into `record.signature.value`. It is not a signature over the Evidence Record body. The Action runtime has no issuer private key, so it cannot sign the record body.

The marker proves only that the producing organization held a valid license for its GitHub owner at runtime. It does not prove that the attempts, weakenings, test files, or workflow content inside the record are true. This is the same honor-system limitation as `owner.type`: an organization with a valid license could alter its own records before aggregation and keep the valid marker. The report says this plainly.

## Organization Evidence Collection

Copy `evidence-repo-template/.github/workflows/collect-evidence.yml` into a dedicated evidence repo, for example `acme/ci-evidence`, and add a `repos.txt` file with one `owner/repo` per line. The workflow runs only on `workflow_dispatch` or `schedule`.

Required secrets in that evidence repo:

| Secret | Purpose |
| --- | --- |
| `EVIDENCE_COLLECT_TOKEN` | Reads Actions artifacts from the listed repositories. |
| `FCP_LICENSE` | Organization license string used to enable aggregation and verify record markers. |
| `FCP_REPORT_SIGNING_KEY` | Organization-owned Ed25519 private key used to sign the report file for tamper detection. |

The workflow downloads `fcp-evidence-*` artifacts and commits the original Evidence Record files to:

```text
evidence-records/<owner>/<repo>/<YYYY>/<PR>-<headSha>.json
```

The committed record directory is the durable source of truth for reaggregation. Artifacts are only the transport medium.

The collection schedule must be shorter than artifact retention. GitHub's default artifact retention is usually 90 days, so weekly collection has a wide margin. If the organization shortens retention below 7 days, weekly collection can lose records; use daily collection or increase artifact retention.

## Aggregate Report Tool

After records are committed, the evidence repo runs:

```bash
npm run build
FCP_LICENSE="$FCP_LICENSE" \
FCP_REPORT_SIGNING_KEY="$FCP_REPORT_SIGNING_KEY" \
node tools/aggregate-report.mjs \
  --input evidence-records \
  --output reports/org-evidence-report.json \
  --markdown reports/org-evidence-report.md
```

`aggregate-report.mjs` includes a record only when all of these are true:

- the copied issuer license signature verifies through the embedded `keyId -> public key` map;
- the license org matches the owner in `record.repo`;
- the license is unexpired at aggregation time;
- the record is within the license's deterministic `maxRepos` cap.

Records without a marker, with an invalid marker, with an owner mismatch, or from repos beyond the lexical `maxRepos` boundary are excluded and counted in the report warnings fields. The report records SHA-256 hashes of every included source record in `provenance.sourceRecordHashes`, then signs the report with the organization's `FCP_REPORT_SIGNING_KEY`.

`integrity.reportSignature` is signed by the organization evidence-collection batch. It is an integrity aid for report and post-aggregation record-set tamper detection. It is not an issuer authenticity guarantee and not a source of audit trust.

## Action Inputs

| Input | Default | Description |
| --- | --- | --- |
| `config-path` | `.github/false-clean-pass.yml` | Optional config file path. |
| `github-token` | | `${{ secrets.GITHUB_TOKEN }}`. Used for diff, Check Run, PR comment. |
| `fail-on` | `error` | `error`, `warning`, or `never`. |
| `sarif-path` | `false-clean-pass.sarif` | Output SARIF 2.1.0 file. |

| `base-test-results-glob` | | Optional baseline JUnit glob. |
| `coverage-summary` | | Coverage summary JSON path. |
| `ci-env-keys` | | Comma-separated env key names only, never values. |
| `requiredJobs` | | Comma-separated required check names to protect against skip-as-success workflow weakening. Keep this synchronized with branch protection. |

| `license` | | Optional C-lite license string. Prefer `env.FCP_LICENSE` from an organization secret. |

Outputs: `result`, `error-count`, `warning-count`, `sarif-path`, `evidence-path`.

## Detection Scope

The guard detects:

- skipped or focused tests;
- empty assertions;
- missing env key declarations;
- ignored failures such as `continue-on-error`, `|| true`, `exit 0`, and `--passWithNoTests`;
- guard weakening, including removing the guard step or lowering `fail-on`;
- coverage threshold and summary drops;
- suppression increases;
- protected baseline changes;
- required job disappearance, required job skip-risk `if:` additions, pull_request trigger narrowing, matrix contraction, and `requiredJobs` narrowing;
- JUnit reports with zero executed tests.

Real merge blocking comes from GitHub branch protection requiring the `false-clean-pass` check. The Action is a reporter and evidence generator.

## Known Limits

- Runtime conditional skips hidden behind app logic or dynamic test generation can evade static detection.
- Forged or selectively copied test result files are not proven authentic.
- External reusable workflow internals and dynamic job names become `mapping_unresolved` warnings instead of hard failures.
- Step-level `if:` conditions are not covered by the required-job detector.
- Direct branch protection changes in the GitHub UI are outside PR diff analysis.
- The license marker does not bind the Evidence Record body. sourceRecordHashes detect changes after aggregation, not pre-aggregation self-attestation changes by a licensed org.

## License Issuing And Key Rollover

Issuer-side license creation is offline and uses no network:

```bash
npm run build
FCP_ISSUER_PRIVATE_KEY="$FCP_ISSUER_PRIVATE_KEY" node tools/issue-license.mjs \
  --org acme \
  --key-id k1 \
  --expires-at 2027-07-01T00:00:00.000Z \
  --max-repos 50
```

`FCP_ISSUER_PRIVATE_KEY` is only for issuer local machines or issuer CI secrets. Customer Action runtimes do not receive it. Never put private key values in repository files, examples, tests, or docs.

Key rollover is release-based. `embeddedPublicKeys` is a `keyId -> public key` map. A new issuer key is introduced by releasing a new `keyId` and issuing new licenses with it; old licenses verify while their old key remains in the map. This is not real-time revocation. Organizations pinned to an old Action version keep that old public-key map until they update.
